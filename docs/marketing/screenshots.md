# Marketing screenshots — index & provenance

app** (the actual Next.js UI talking to the real API and a seeded local
Supabase), showing obviously-demo data for **"Mike's Plumbing"**. No mockups, no
stock, no invented capabilities. This file is the honest record of what exists,
how it was made, and how to place it.

Regenerate everything with `apps/web/scripts/capture-shots.mjs` (see
[Reproduce](#reproduce) below). The script is re-runnable and idempotent.

## Manifest API (`apps/web/public/shots/manifest.ts`)

Autogenerated, typed, and typechecked (it is inside the web tsconfig `include`).
Do not hand-edit — re-run the capture script.

```ts
export type Shot = {
  src: string;         // "/shots/<key>.webp"
  avif: string;        // "/shots/<key>.avif" (same display size)
  width: number;       // intrinsic display width (px)
  height: number;      // intrinsic display height (px)
  placeholder: string; // base64 WebP data-URI, ~20px wide (blur-up)
  alt: string;         // honest description of the real screen
  theme: "light" | "dark";
  frame: "browser" | "phone";
};

export const shots: Record<string, Shot>;   // keyed by "<surface>-<theme>"
export type ShotKey = keyof typeof shots;
export function shotPair(id: string): { light?: Shot; dark?: Shot };
```

**Placement recipe (keeps Lighthouse ~100, VISUALS §5):**

- Wrap in `<Frame variant="browser" | "phone">` (VISUALS §4.3) — browser chrome
  for desktop shots, a phone frame for mobile shots (the `frame` field tells you
  which). Add the single petrol glow + 1px border + soft shadow per VISUALS §1A.
- Render a `<picture>`: `<source type="image/avif" srcSet={shot.avif}>` then the
  WebP `<img src={shot.src}>`.
- Always set `width={shot.width} height={shot.height}` (zero CLS). Add
  `loading="lazy"` + `decoding="async"` for anything below the fold; use
  `shot.placeholder` as the blur-up background behind the image while it loads.
- Use `shotPair("inbox-list")` to grab the light+dark pair for a surface and
  render the theme-correct one (or both, toggled by the active theme).
- The H1 remains the LCP element — never let a screenshot become the LCP.

**Placement tracks (where each shot is meant to go, per VISUALS §3):**

- `inbox-list` → home first scroll-reveal (framed inbox), inbox feature page hero.
- `thread-open` → home "the conversation is the hero" section; shared-inbox /
  thread feature page hero; the nav Product mega-menu featured cell thumbnail.
- `contact-panel` → contacts/CRM feature section.
- `mobile-inbox` + `mobile-thread` → the dark PWA / "works in the truck" band
  (phone-framed), mobile-first proof on the home body.
- `onboarding-setting-up` → the "how it works" / "get your number" section and
  the onboarding-timeline surfaces (the number-reveal is the activation moment).

## Reproduce

Prerequisites (all local): Docker + Supabase CLI, Node ≥22, pnpm, and the
Playwright chromium binary. The web app **must** run on port **3100** — the API's
`APP_ORIGIN` CORS allowlist is an exact match.

```bash
# 1. Local stack
pnpm db:start                 # once
pnpm db:reset                 # clean migrations state

# 2. Seed the Mike's Plumbing demo data (service key)
node --experimental-strip-types scripts/dev-seed.ts

# 3. API worker (real .dev.vars from .dev.vars.example — local Supabase + fakes)
pnpm --filter @loonext/api exec wrangler dev --port 8787

# 4. Web app (in another shell; .env.local → local Supabase + the wrangler API)
pnpm --filter @loonext/web exec next dev --port 3100

# 5. Chromium for Playwright (once)
pnpm dlx playwright install chromium

# 6. Capture (from the repo root)
node apps/web/scripts/capture-shots.mjs
```

The script signs in as `owner@loonext.test` / `devseed1`, drives each surface,
captures 2× light+dark, downscales to WebP+AVIF at the display width, and
rewrites `manifest.ts`. Raster work happens in `.next-shots/` (gitignored via
`.next-*/`) and is cleaned up on exit. **Kill the two dev servers yourself when
done** — the capture script does not own them.

Override the target with `SHOTS_BASE_URL` if the web app runs elsewhere (its
origin must still equal the API's `APP_ORIGIN`).

## Honesty & gaps

**Real vs. mocked:** every file under `public/shots/` is a real capture of the
running product. Nothing here is fabricated.

**MMS image note:** the two inbound "job photos" are procedurally generated
placeholder PNGs (a gray under-sink scene) uploaded to the real `mms-media`
bucket and fetched through the real signed-URL path — i.e. the *product feature*
(inbound MMS, thumbnail, blur-up, lightbox) is shown truthfully; only the
photograph's subject is synthetic demo content, not a real customer photo. This
is honest demo data, not a faked capability.

them — capture in a follow-up rather than fabricate):

  first-message footer preview, quiet-hours dialog).
- A **settings / usage meter** screen (the big period meter with the 80% amber
  state).
- A **web-push notification** (contact name + snippet) — needs an OS-level
  notification capture, out of scope for the headless Playwright pass.
- The **sole-prop OTP** onboarding row and the **registration-pending amber
  banner** above the inbox (the seed uses an already-approved company; a second
  seed variant would be needed to show the pending state honestly).

pending/registration states), add a shot entry to `SHOTS` in
`capture-shots.mjs`, and re-run. The manifest and this table regenerate.
